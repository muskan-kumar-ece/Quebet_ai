### FEATURE 04 — Hardware Risk Score Engine
**Category:** Analysis  
**What it does:** The platform's signature feature. Accepts full circuit description (components, topology, operating conditions). Four subsystems score independently: Thermal Analysis (30%), EMI Risk (20%), Design Quality (30%), Component Quality (20%). Scores are aggregated into a 0–100 risk index. Output is an animated dial + downloadable PDF report.

**Input → Output Flow:**
```
User describes circuit or uploads circuit JSON
  → POST /analysis/risk-score
    → services/analysis/risk_scorer.py
      → Runs 4 parallel evaluators:
          failure_predictor (design rules check)
          power_thermal (NumPy thermal math)
          emi_analyzer (frequency domain checks)
          component quality (cross-check against components DB)
      → Each returns partial score + findings[]
        → Aggregator: weighted_score = Σ(weight × subscore)
          → LLM receives findings + prompts/analysis/risk_score.py
            → Returns JSON: { score, factors[], recommendations[] }
              → RiskScoreMeter.tsx animates from 0 → final score
```

**Files:** `services/analysis/risk_scorer.py`, `prompts/analysis/risk_score.py`, `components/shared/RiskScoreMeter.tsx`, `components/modules/analysis/RiskReport.tsx`

**Value:** Hardware engineers spend hours in design review catching issues. This flags critical problems before a PCB is ordered — preventing $500–$5000 re-spins.

### FEATURE 06 — Power & Thermal Analyzer
**Category:** Analysis  
**What it does:** Accepts a component list with power dissipation values and ambient temperature. NumPy calculates junction temperatures using thermal resistance (Rθja) from component specs. LLM interprets results and suggests heatsink sizes, airflow requirements, and component derating.

**Input → Output Flow:**
```
User provides component list (each with Pd, Rθja, ambient temp)
  → POST /analysis/thermal
    → services/analysis/power_thermal.py
      → NumPy: Tj = Ta + (Pd × Rθja) for each component
        → Identify components exceeding max Tj
          → LLM receives thermal data + prompts/analysis/power_thermal.py
            → Returns cooling suggestions, derating recommendations
              → ThermalHeatmap.tsx renders color gradient overlay on component positions
```

**Files:** `services/analysis/power_thermal.py`, `prompts/analysis/power_thermal.py`, `components/modules/analysis/ThermalHeatmap.tsx`

**Value:** Thermal failures account for ~55% of electronic equipment failures. Engineers without simulation tools rely on gut feel — this provides physics-based analysis + engineering judgment.

### FEATURE 20 — LLM Provider Switching (Claude ↔ OpenAI)
**Category:** System / Infra  
**What it does:** Every AI call in the platform goes through `provider_factory.py`. Setting `AI_PROVIDER=claude` routes all completions and vision calls to Anthropic's API. Setting `AI_PROVIDER=openai` switches to GPT-4o. Zero code changes required. Users can even select their preferred AI in Settings → the choice is stored per-user and sent in the request.

**Input → Output Flow:**
```
User selects "Claude" in Settings → saved to aiStore.ts
  → Every API request includes X-AI-Provider: claude header
    → Next.js BFF /api/proxy/* forwards header
      → FastAPI dependencies.py: get_ai_provider() reads header
        → provider_factory.py returns ClaudeProvider() or OpenAIProvider()
          → Both implement BaseProvider ABC: complete(), stream(), analyze_image()
            → Service calls provider.complete(system, user) — same call regardless
```

**Files:** `services/ai/provider_factory.py`, `services/ai/claude_provider.py`, `services/ai/openai_provider.py`, `services/ai/base_provider.py`, `core/dependencies.py`, `stores/aiStore.ts`

**Value:** Anthropic API outage? Switch to OpenAI in one env var change. Cost optimization: use Claude for complex reasoning, GPT-4o for vision — route by task type without any refactor.

### FEATURE 21 — Async Job Queue with Live Progress
**Category:** Infra  
**What it does:** Heavy AI tasks (PCB image analysis: 15s, full simulation: 30s) never block HTTP. POST request returns `{ job_id }` in 200ms. Celery worker picks up the task from Redis queue. `JobStatusPoller.tsx` polls `/jobs/{job_id}` every 2 seconds showing a live progress bar. When complete, result renders automatically.

**Input → Output Flow:**
```
User triggers heavy task (PCB analyze, simulation)
  → POST /analysis/... → returns { job_id: "abc123" } in <200ms
    → JobStatusPoller.tsx starts polling GET /jobs/abc123 every 2s
      → FastAPI: job_repo.get_result(job_id) → returns { status: "running", progress: 60 }
        → Celery worker (worker/tasks/) completes task → job_repo.update_status(done, result)
          → Next poll returns { status: "done", result: {...} }
            → Frontend renders result, stops polling
```

**Files:** `worker/celery_app.py`, `worker/tasks/ai_tasks.py`, `api/v1/routes/jobs.py`, `repositories/job_repo.py`, `components/shared/JobStatusPoller.tsx`, `hooks/useJobPoller.ts`

**Value:** Without async handling, a 30-second LLM call would timeout in most web frameworks. This pattern scales to thousands of concurrent users without blocking any thread.

### FEATURE 22 — Prompt-Level Redis Cache
**Category:** Infra  
**What it does:** Every LLM call is hash-keyed by (model + system_prompt + user_prompt). Before calling the Anthropic/OpenAI API, `response_cache.py` checks if this exact prompt combination exists in Redis. If yes — returns cached response in <10ms, LLM API not called. TTL = 3600s. Saves 80%+ on API costs for repeated queries.

**Input → Output Flow:**
```
Service calls provider.complete(system, user)
  → response_cache.py: cache_key = SHA256(model + system + user)
    → Redis GET cache_key
      → HIT: return cached_response immediately (skip LLM)
      → MISS: call LLM → get response → Redis SET cache_key response TTL=3600
        → Return response
```

**Files:** `services/ai/response_cache.py`, `core/redis_client.py`, `utils/cache.py`

**Value:** The same datasheet simplification question gets asked by 1000 students. Without caching: 1000 × $0.02 = $20 in API costs. With caching: 1 call + 999 cache hits = $0.02.

### FEATURE 23 — BFF API Gateway with Auth Injection
**Category:** Infra  
**What it does:** The Next.js `/api/proxy/[...path]` route acts as a Backend-for-Frontend. The browser never knows the FastAPI URL. The BFF validates the NextAuth session, injects the JWT as `Authorization: Bearer`, applies per-user rate limiting, and forwards to FastAPI. API keys never leave the server.

**Input → Output Flow:**
```
Browser calls: POST /api/proxy/analysis/risk-score  (no auth header needed)
  → Next.js route.ts (app/api/proxy/[...path]/route.ts)
    → Validates NextAuth session server-side
      → Applies rate limiting (Redis counter: user_id:endpoint:minute)
        → Injects: Authorization: Bearer {jwt_token}
          → Forwards to FastAPI: POST http://backend:8000/api/v1/analysis/risk-score
            → Returns response to browser
```

**Files:** `app/api/proxy/[...path]/route.ts`, `app/api/auth/[...nextauth]/route.ts`, `lib/api.ts`, `lib/auth.ts`

**Value:** Directly calling FastAPI from the browser exposes backend URLs, skips server-side auth validation, and puts API keys at risk. BFF pattern is non-negotiable for production security.

### 4.1 What Data Gets Embedded

The RAG corpus has five data domains, each with different embedding strategies:

```
CORPUS DOMAINS:
├── circuits/               → reference-designs.json + common-topologies.json + user-generated circuits
├── failure_cases/          → thermal.json + emi.json + component-failures.json + user-reported failures
├── components/             → resistors.json + capacitors.json + ics.json + indian-market.json
├── datasheets/             → extracted text from PDF datasheets (chunked by section)
└── user_projects/          → saved project descriptions + analysis results (per-user)
```

**What gets embedded per domain:**

| Domain | Embedding Unit | Why This Granularity |
|---|---|---|
| Circuits | Full circuit description (topology + component list) | Full context needed for similarity |
| Failure cases | Symptom text + root cause text (separate embeddings, linked by ID) | Symptom search ≠ cause search |
| Components | Name + specs + application notes (concatenated) | "What's a good cap for switching PSU?" needs all three |
| Datasheets | 512-token chunks with 50-token overlap | Datasheets are long; chunked retrieval beats full-doc |
| User projects | Circuit description at save time | Personalized RAG: your past designs inform future designs |

**Chunking Strategy by Document Type:**

```python
# Datasheets: Section-aware chunking (smarter than naive 512-token splits)
def chunk_datasheet(text: str) -> list[Chunk]:
    sections = split_by_headers(text)  # "Electrical Characteristics", "Timing", etc.
    chunks = []
    for section in sections:
        if len(section.tokens) <= 512:
            chunks.append(Chunk(text=section.text, metadata={"section": section.title}))
        else:
            # Sliding window with 50-token overlap for long sections
            chunks.extend(sliding_window(section.text, window=512, overlap=50))
    return chunks

# Circuit descriptions: No chunking — embed whole description
# Preserves the relational context between components
def embed_circuit(circuit: dict) -> Embedding:
    text = f"Circuit: {circuit['name']}. Topology: {circuit['topology']}. "
           f"Components: {', '.join(c['name'] for c in circuit['components'])}. "
           f"Purpose: {circuit['description']}."
    return embed(text)
```

### 4.3 ChromaDB Usage

**Collections Structure:**
```
ChromaDB Collections:
├── electronics_kb          (static, seeded at startup, ~10K vectors)
├── datasheets              (grows as users upload PDFs, ~100K vectors at scale)
├── user_projects           (per-user namespace via metadata filter)
```

**Similarity Search → Context Injection:**
```python
# In services/knowledge_graph/embeddings.py
async def retrieve_relevant_context(query: str, domain: str, user_id: str, k: int = 5) -> str:
    query_embedding = await embed(query)
    
    # Domain-specific search with metadata filter
    results = chroma_client.get_collection(domain).query(
        query_embeddings=[query_embedding],
        n_results=k,
        where={"$or": [
            {"domain": domain},
            {"user_id": user_id}  # Include user's own past projects
        ]}
    )
    
    # Format for LLM context injection
    context_blocks = []
    for doc, metadata, distance in zip(results['documents'][0], results['metadatas'][0], results['distances'][0]):
        relevance_score = 1 - distance  # Cosine distance → similarity
        if relevance_score > 0.7:  # Threshold: only inject high-quality matches
            context_blocks.append(f"[Relevance: {relevance_score:.2f}] {doc}")
    
    return "\n---\n".join(context_blocks)
```

**Context Injection into LLM Prompt:**
```python
# In services/design/project_builder.py
async def build_project(idea: str, board: str, user_id: str) -> ProjectOutput:
```

### 4.4 Hybrid Retrieval: Vector Search + Neo4j Graph

The platform uses a two-stage retrieval architecture. Vector search finds semantically similar content. Graph queries find structurally related content. Together they cover both types of knowledge:

```
USER QUERY: "ESP32 WiFi circuit is overheating"

STAGE 1 — VECTOR SEARCH (ChromaDB)
  → Finds semantically similar failure cases:
      "ESP32 thermal issue in compact enclosure" (score: 0.91)
      "WiFi module power consumption spike" (score: 0.87)
      "Microcontroller brownout from thermal throttling" (score: 0.83)

STAGE 2 — GRAPH QUERY (Neo4j)
  → Cypher: MATCH (c:Component {name: "ESP32"})-[:HAS_FAILURE]->(f:Failure)
             WHERE f.category = "thermal"
             RETURN f.description, f.fix ORDER BY f.frequency DESC
  → Returns: 
      "ESP32 running at 240MHz with WiFi active draws 240mA peak. 
       USB power without bulk decoupling causes brownout."
       Fix: "Add 100µF bulk cap at 3.3V rail + heatsink pad to GND pour"

STAGE 3 — RANK + MERGE
  → Re-rank combined results by: relevance_score × recency × community_upvotes
  → Top 3 results injected into LLM context

STAGE 4 — LLM GENERATION
  → Grounded response with specific values, component suggestions, fixes
```

**Why Both Are Needed:**

| Query Type | Vector Search | Graph Query |
|---|---|---|
| "Find circuits like mine" | ✅ Semantic similarity | ❌ Can't find structural analogs |
| "What fails with ESP32?" | ❌ May miss exact component | ✅ Exact node traversal |
| "Fix for thermal + WiFi issue" | ✅ Finds related text | ✅ Finds historical fixes |
| "Which component causes this symptom?" | ❌ Weak | ✅ Symptom→Failure→Component path |

### 4.6 Caching Strategy

```
THREE-LEVEL CACHE HIERARCHY:

L1 — PROMPT CACHE (Redis, TTL: 1 hour)
  Key: SHA256(model + system_prompt + user_prompt)
  What: Exact LLM responses for identical queries
  Hit rate: ~40% (common questions: "what is Ohm's law", "simplify 555 timer datasheet")
  Savings: 40% reduction in LLM API costs

L2 — EMBEDDING CACHE (Redis, TTL: 24 hours)
  Key: SHA256(text_to_embed)
  What: Embedding vectors for recently embedded texts
  Prevents: Re-embedding the same component description 1000 times
  Hit rate: ~70% (same components appear in many queries)

L3 — SEARCH RESULT CACHE (Redis, TTL: 15 minutes)
  Key: SHA256(collection + query_embedding + n_results + filters)
  What: ChromaDB query results for recent searches
  Prevents: Vector DB round-trip for identical retrieval queries
  Hit rate: ~25% (user sessions often refine the same query)

INVALIDATION STRATEGY:
  - New circuit added to KB → invalidate L3 for electronics_kb collection
  - User updates project → invalidate L2 + L3 for user_projects
  - Prompt template changed → invalidate L1 for that prompt hash prefix
```

### 5.1 LLM Provider Abstraction

The `BaseProvider` ABC is the cornerstone of the entire AI layer. It defines a contract that any LLM must fulfill:

```python
# services/ai/base_provider.py
class BaseProvider(ABC):
    @abstractmethod
    async def complete(self, system: str, user: str, temperature: float = 0.1) -> str:
        """Text completion — returns raw string"""
    
    @abstractmethod
    async def stream(self, system: str, user: str) -> AsyncGenerator[str, None]:
        """Streaming completion — yields tokens"""
    
    @abstractmethod
    async def analyze_image(self, image_url: str, prompt: str) -> str:
        """Vision analysis — returns description"""
    
    @abstractmethod
    async def embed(self, text: str) -> list[float]:
        """Text embedding — returns vector"""

# Task-specific routing (advanced optimization):
class ProviderRouter:
    def get_provider_for_task(self, task: TaskType) -> BaseProvider:
        routes = {
            TaskType.VISION: self.claude,       # Claude vision is stronger
            TaskType.LONG_CONTEXT: self.claude,  # Claude 200K context window
            TaskType.FAST_COMPLETION: self.openai,  # GPT-4o-mini for simple tasks
            TaskType.EMBEDDING: self.openai,     # text-embedding-3-small is cost-effective
        }
        return routes.get(task, self.default_provider)
```
